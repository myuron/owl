# owl — 開発ガイド

vim ライクな GTK4/WebKitGTK 6 ベースのブラウザ(Rust)。要求は `docs/requirements.md`、
設計は `docs/design.md`、テスト項目は `docs/test.md`、実装手順は `docs/todo.md`
(進捗は `docs/checklist.md`)にある。**着手前にこれらを読む。**

## コマンド

`Justfile` 経由で回す。**GTK/WebKit ネイティブ依存のため `just` 系はすべて `nix develop` 内で実行する**(CI も `nix develop --command just ci`):

- `just test` — `cargo test`(GTK 不要の純粋ロジックのみで完結すること)
- `just lint` — `cargo clippy -- -D warnings`(警告ゼロ必須)
- `just coverage` — `cargo llvm-cov` で純粋ロジックの region/line カバレッジを **100%** に強制
  (`main.rs`/`window.rs`/`webview.rs` の GTK 依存は除外)。未テストの `match` アーム(=通らない region)を機械検出するゲート。
- `just mutants` — `cargo-mutants` で純粋モジュール(`command.rs`/`keys.rs`)に限定した
  ミューテーションテスト。**coverage が見ない「分岐内の挙動」の未検証(丸めを truncate に
  変えても通る等、規約 4)を機械検出する**上位ゲート。survivor が出たらテストを足す。
- `just fmt-check` / `just fmt` — フォーマット確認 / 適用
- `just ci` — `fmt-check → lint → coverage → mutants → build` を通す(PR 前に必ず緑にする)

## ワークフロー

- **TDD**(`docs/todo.md` のサイクル単位): Red(先にテスト→コンパイルエラー確認)→
  Green(最小実装)→ Refactor(緑を保ったまま整理)。各ステップ後に `cargo test`。
- **コミットは 1 機能単位(`design.md` の該当節。例: M7 なら §8.2〜§8.6)で切る。マイルストーン
  単位で丸めない。** `docs/todo.md` のサイクル 3 以降は 1 サイクル = 1 マイルストーン相当なので
  「サイクル = コミット」にすると巨大化する(悪例: `41687f9` は Cookie 永続化・TLS Fail・ポップアップ抑制・DL キャンセル・
  エラーページ/クラッシュ復帰の 6 機能を 1 コミットに混載)。**1 機能の Red/Green/Refactor(または
  `-B` 結線 1 機能分)= 1 コミット**とし、複数機能を混ぜない。コミットメッセージは機能と出典節を
  特定できる形にする(例: `feat(m7): エラーページ HTML 生成 (§8.6, todo 9-A)` /
  `feat(m7): Cookie 永続化を結線 (§8.2, todo 9-B)`)。M1/M2 が `todo 3-4`・`cycle 4` を参照した
  precedent に倣う。`docs/todo.md` の該当ステップ(`9-A`/`9-B` 等)のチェックボックスは、その機能を
  実装したコミットと同一コミットで更新する(後述のドキュメント整合ルールと同型)。
- **純粋ロジックと GTK を分離する。** テスト対象(`command.rs` の `parse_open_input`、
  `keys.rs` の `resolve_key` 等)は gtk/webkit クレートに依存させない。副作用(実スクロール・
  クリップボード・DOM 操作・ステータスバー更新)は呼び出し側が担う。
- **最初にファイルを変更する前に、`main` から新しいブランチを切る。** どのファイルであれ
  最初の編集に着手する前段でブランチを作成し、以降の作業はその上で行う(コミット/プッシュ・
  PR 作成はユーザーの指示で行う)。既にトピックブランチ上にいるならそのまま使ってよい。
- **実装が `design.md` と食い違ったら、同一 PR で `design.md` を更新する。** マイルストーンの
  再スコープ(例: M2 のナビゲーションを M3 へ移動)・前倒し(例: 引数 URL を M1 へ)・技術判断の
  変更は、`docs/todo.md`/`checklist.md` だけでなく**権威ドキュメントである `design.md`(§16 等)へ
  必ず反映する**(M1 が §17 を更新した precedent に倣う)。`docs/todo.md` の完了条件チェックボックスは
  実態と一致させ、同一コミットで更新する。
- **`main` へのマージは勝手にしない。** マージはユーザーが明示的に指示したときだけ実行する
  (`gh pr merge` を自己判断で叩かない)。基本フローは「ブランチ → 機能ごとに複数コミット → push → PR 作成」で
  止める(PR は従来どおり 1 マイルストーン = 1 PR。粒度を細かくするのはコミットだけで、ブランチ/PR/`/review` は不変)。
- **PR を作成したら、`/review` をサブエージェントで実施する。** `gh pr create` の直後に、
  Agent ツールで**最上位モデル(`model: fable` — Claude Fable 5)**のサブエージェントを起動し、
  対象 PR 番号を渡して `/review` を回す。レビュー結果を取得してユーザーに要約提示する(マージはしない)。

## コーディング規約

- **doc コメントは日本語**で、根拠となる設計書の節(`§7.3` 等)を必ず引用する
  (`src/command.rs` を範とする)。テスト名は仕様 ID を接頭辞にする(`p01_…` / `k10_…`)。
- 未結線で dead_code になるモジュールは、冒頭に理由を書いて `#![allow(dead_code)]`。

### コードレビューで繰り返し出た指摘 — 先回りで潰す

これらは単発バグではなく**再発する型**。実装・レビュー時に必ず確認する:

1. **スコープ外の `match` アームを手近な値で埋めない。** 仕様表(例: `design.md §7.2` の
   モード別挙動)にある分岐は、実装しないなら**仕様に沿った安全側**へ倒すか `todo!()` にし、
   **その挙動をテストで固定する**。「とりあえず `proceed()`」のような誤挙動の固定化を避ける。
   - 悪例: Hint モードを `Command` と同じ `proceed()` にする(§7.2 は Stop が正)。

2. **相異なる `Action`/`Propagation` を返す `match` アームは、一つ残らずテストを持つ。**
   テスト対象を `test.md` の ID に閉じない。ID に無い分岐(例: `Insert + Esc`、`Command`/`Hint`
   の素通し)も、独立したコードパスなら必ずテストする。

3. **public API に使わない引数を足さない。** 「常に同じ値を返す」「守るべき不変条件がある」なら、
   関数の doc コメントではなく**型(struct)で強制**する。
   - 悪例: `set_mode(_pending, mode) -> (mode, None)` — 第1引数を無視し、不変条件が
     コメントにしかない。状態を持つなら `struct` + メソッドにして不変条件を型で表す。
   - **時間的不変条件(「起動時 1 回だけ呼ぶ」等)もコメントで担保しない。** 呼び出しを
     「1 回」が構造的に真になる場所(起動側)へ移すか、`std::sync::Once` 等で強制する。
     - 悪例: per-window の関数から `install_css` を呼び、「`NON_UNIQUE` なので 1 回」を
       doc コメントにだけ書く(2 ウィンドウ化・再 `activate` で静かに壊れる)。

4. **`test.md` の 1 ID は期待結果の全節をアサートする。** 「クリアされる」だけでなく
   「クリア後に次のキーが新規シーケンスとして扱われる」まで、仕様の記述を余さず検証する。
   - **`just coverage` の 100% は「分岐の網羅」であって「挙動の網羅」ではない。** doc/仕様が
     具体挙動(丸め・エンコード・順序・クランプ等)を主張するなら、**誤実装でも同じ region を
     通る**ため coverage では検出できない。**誤実装で落ちる値を最低 1 つ**テストに選ぶ。
     - 悪例: 丸めの検証に `0.42`(round でも truncate でも `42%`)だけを使う。
       正: `0.999`(round=`100%`、truncate=`99%`)を足し、`round()` を truncate に
       変えたら落ちるようにする。この型は `just mutants`(cargo-mutants)でも機械検出する。

5. **仕様表を実装するときは全行を網羅する。** 未対応の行(例: 修飾キー `Ctrl+d`/`Ctrl+u`)は、
   黙って落とさず「未実装」と分かる形(TODO コメント/型で表現不能なら doc に明記)にする。

> **6〜9 は「coverage/mutants が見ない結線層」の型。** `input.rs` の GTK 結線・`page.js` は
> 自動テストが無く(§14)、これらの型はレビューでしか捕まらない。実装時のセルフレビューで
> 必ず 6〜9 を通し、PR 後の `/review`(§ワークフロー)もこの層を主対象にする。

6. **信頼境界を越える入力は「現在の状態」で検証してから作用する。** JS → Rust(script message
   handler)は main world・全フレームに公開され、**任意ページ(クロスオリジン iframe 含む)が
   偽装できる**。Rust 側ハンドラは「そのメッセージが現在のモード/状態で妥当か」を必ず確認し、
   想定外なら黙って無視する(安全側)。ページを信頼しない。
   - 悪例: `hint_result` を現モード無視で適用し、任意ページが owl を Insert へ強制遷移できる
     (§9・要求 3.3 の「スクリプト起因では Insert に入らない」に反する)。
   - 正: `input::install_hint_message_handler` は現モードが `Hint` のときのみ hint 結果を受理する。

7. **非同期の副作用に先行依存してモード遷移したら、「応答が来ない経路」を必ず考える。** Rust が
   先にモードを変えて JS を fire-and-forget で駆動する場合、JS が応答しない(未注入・Web プロセス
   クラッシュ・`window.owlHints` の削除/上書き・hint 表示中のページ遷移)とスタックしうる。
   最低限 `Esc` 等の脱出路を保証し、既知の制限を `design §17`・checklist に明記する。
   - 悪例: `f` で `Hint` へ遷移後、`hint_none`/`hint_result` が来ず `-- HINT --` のまま(全キー Stop)。

8. **固定容量のジェネレータを可変長ループで消費しない。** 生成側の上限(例: ラベルは
   `14 + 14×14 = 196` 個)を消費側(要素数)が超えると undefined/範囲外アクセス → 例外/panic に
   なる。消費側を生成数へクランプするか生成側を必要数へ拡張する。**さらにその失敗がユーザーを
   詰まらせないこと**(例外で overlay 未表示かつ通知も飛ばず Hint スタック、等)まで確認する。
   - 悪例: `for (j < elements.length)` で `labels[j]` を触り、196 要素超のページで TypeError。

9. **手動確認(checklist)の期待結果は「設計の理想」でなく「実装の実挙動」を書く。** 実装を追い、
   実際に起きること(例: `f` 直後は一瞬 `-- HINT --` になる)を記述する。理想を書くと**正しい挙動を
   バグと誤判定/実バグを見逃す**。実装が理想と食い違うなら、まず実装の是非を判断してから記述する。
   - 悪例: about:blank で `f` を「Normal のまま」と書く(実際は一瞬 `-- HINT --` → `hint_none`)。
