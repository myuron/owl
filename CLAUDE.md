# owl — 開発ガイド

vim ライクな GTK4/WebKitGTK 6 ベースのブラウザ(Rust)。設計は `docs/design.md`、
テスト項目は `docs/test.md`、実装手順は `docs/todo.md` にある。**着手前にこれらを読む。**

## コマンド

`Justfile` 経由で回す(CI は `just ci` を実行する):

- `just test` — `cargo test`(GTK 不要の純粋ロジックのみで完結すること)
- `just lint` — `cargo clippy -- -D warnings`(警告ゼロ必須)
- `just fmt-check` / `just fmt` — フォーマット確認 / 適用
- `just ci` — `fmt-check → lint → test → build` を通す(PR 前に必ず緑にする)

## ワークフロー

- **TDD**(`docs/todo.md` のサイクル単位): Red(先にテスト→コンパイルエラー確認)→
  Green(最小実装)→ Refactor(緑を保ったまま整理)。各ステップ後に `cargo test`。
- **純粋ロジックと GTK を分離する。** テスト対象(`command.rs` の `parse_open_input`、
  `keys.rs` の `resolve_key` 等)は gtk/webkit クレートに依存させない。副作用(実スクロール・
  クリップボード・DOM 操作・ステータスバー更新)は呼び出し側が担う。
- 作業は `main` から新しいブランチを切り、PR を作る(コミット/プッシュはユーザーの指示で行う)。

## コーディング規約

- **doc コメントは日本語**で、根拠となる設計書の節(`§7.3` 等)を必ず引用する
  (`src/command.rs` を範とする)。テスト名は仕様 ID を接頭辞にする(`p01_…` / `k10_…`)。
- 未結線で dead_code になるモジュールは、冒頭に理由を書いて `#![allow(dead_code)]`。

### コードレビューで繰り返し出た指摘 — 先回りで潰す

これらは単発バグではなく**再発する型**。実装・レビュー時に必ず確認する:

1. **スコープ外の `match` アームを手近な値で埋めない。** 仕様表(例: `design.md §7.2` の
   モード別挙動)にある分岐は、実装しないなら**仕様に沿った安全側**へ倒すか `todo!()` にし、
   **その挙動をテストで固定する**。「とりあえず `proceed()`」のような誤挙動の固定化を避ける。
   - 悪例: Hint モードを `Command` と同じ `proceed()` にする(§7.2 は Stop が正)。

2. **相異なる `Action`/`Propagation` を返す `match` アームは、一つ残らずテストを持つ。**
   テスト対象を `test.md` の ID に閉じない。ID に無い分岐(例: `Insert + Esc`、`Command`/`Hint`
   の素通し)も、独立したコードパスなら必ずテストする。

3. **public API に使わない引数を足さない。** 「常に同じ値を返す」「守るべき不変条件がある」なら、
   関数の doc コメントではなく**型(struct)で強制**する。
   - 悪例: `set_mode(_pending, mode) -> (mode, None)` — 第1引数を無視し、不変条件が
     コメントにしかない。状態を持つなら `struct` + メソッドにして不変条件を型で表す。

4. **`test.md` の 1 ID は期待結果の全節をアサートする。** 「クリアされる」だけでなく
   「クリア後に次のキーが新規シーケンスとして扱われる」まで、仕様の記述を余さず検証する。

5. **仕様表を実装するときは全行を網羅する。** 未対応の行(例: 修飾キー `Ctrl+d`/`Ctrl+u`)は、
   黙って落とさず「未実装」と分かる形(TODO コメント/型で表現不能なら doc に明記)にする。
