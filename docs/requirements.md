# owl — 要求文書

## 1. 概要

owl は、WebKitGTK を組み込んだキーボード主体のミニマルな個人用 Web ブラウザである。Rust で実装し、qutebrowser のような vim 風の操作体系を第一級の機能として設計する。レンダリングエンジンは自作せず、既存エンジン(WebKitGTK)に任せることで、日常の Web 閲覧に耐える互換性を早期に確保する。

## 2. 背景・目的

- 作者(myuron)が日常使いする実用ブラウザを自分の手で作りたい。学習だけが目的ではなく、完成後は主用途のブラウザとして常用することを目指す。
- 既存ブラウザへの不満は「マウス操作を前提とした UI」と「機能過多」。owl は全操作がキーボードだけで完結することと、必要最小限の機能に絞ったミニマリズムを設計原則とする。マウス操作(クリック・ホイールスクロール)を無効化はしないが、マウスがなければ使えない操作を作らない。
- レンダリング・JavaScript 実行はエンジン(WebKitGTK / JavaScriptCore)に委譲し、owl 自体の開発対象は「ブラウザの殻」(ウィンドウ、キーバインド、ナビゲーション、コマンドライン)に集中する。

## 3. スコープ(MVP)

MVP は「owl だけで日常的なページ閲覧が一通りできる」状態を目標とする。

### 3.1 ページ表示

- 単一ウィンドウで Web ページを表示できる。HTML/CSS のレンダリングと JavaScript の実行は WebKitGTK(JavaScriptCore 同梱)に委譲する。
- HTTPS サイトを閲覧できる。証明書が無効なサイトは読み込みをブロックしてエラー表示し、MVP では続行(例外許可)の手段は提供しない。
- ページ側が新規ウィンドウを要求した場合(`target="_blank"`、`window.open`)は、新規ウィンドウを開かず同一ウィンドウ(現在の WebView)で開く。既知の制限: ポップアップから `postMessage` で opener に結果を返す型の OAuth ログインはこの方式では成立しない。リダイレクト型フローで代替できないサービスで支障が出た場合は、例外的な新規ウィンドウ許可を検討する(この文書の改訂事項)。
- 読み込みエラー(DNS 解決失敗等)時はエラーページを表示する。レンダラプロセス(WebProcess)がクラッシュした場合も白画面のまま放置せず、エラー表示とリロードによる復帰ができること。
- Cookie・ローカルストレージ等のサイトデータは XDG データディレクトリ配下に**永続化**する。ログイン状態が再起動をまたいで保持されることは日常使いの必須条件である。
- JS ダイアログ(alert/confirm/prompt)や HTTP Basic 認証は、エンジン/GTK の既定ダイアログをそのまま許容する(ミニマリズムの例外とする)。
- ファイルダウンロードは MVP では**非対応**とする。WebKitGTK は既定ではダウンロード要求を `~/Downloads` へ自動保存するため、owl はダウンロード開始を検知して能動的にキャンセルする(黙って保存させない)。対応自体は将来拡張に送る。

### 3.2 ナビゲーション

- URL を指定してページを開ける。
- 戻る / 進む / リロード / 読み込み中断ができる(操作は 3.3 のキーバインドに割り当てる)。
- 現在の URL とページタイトル、読み込み状態、現在のモード(vim の `-- INSERT --` 相当のインジケータ)をステータスバーに表示する。
- 起動時はコマンドライン引数で渡された URL(`owl https://example.com`)を開く。引数がなければ空白ページを表示する。

### 3.3 vim 風キーバインド

owl はモーダル UI を採る: キー入力は現在のモードに応じて「owl へのコマンド」か「ページへの入力」のどちらかに解釈され、これによりマウスなしで閲覧操作とテキスト入力を両立する。操作体系は qutebrowser を参考にする。

- **normal モード**(デフォルト): キー入力を owl のコマンドとして解釈する。以下に列挙しない修飾なしキーは無視し、ページへは渡さない。修飾キー付き入力(選択テキストの `Ctrl+C` コピー等)は、owl が明示的にバインドするものを除きページへ素通しする。
  - `h` / `j` / `k` / `l`: スクロール(左 / 下 / 上 / 右)。1 回のスクロール量は実装裁量(目安: 50px 程度)
  - `gg` / `G`: ページ先頭 / 末尾へ移動
  - `Ctrl+d` / `Ctrl+u`: 半ページ(ビューポート高の 1/2)下 / 上スクロール
  - `H` / `L`: 履歴の戻る / 進む
  - `r`: リロード
  - `yy`: 現在ページの URL をクリップボードへコピー
  - `Esc`: 読み込み中断
  - `f`: hint モードへ
  - `i`: insert モードへ
  - `:`: command モードへ
- **insert モード**: キー入力をページ(フォーム等)へ素通しする。`Esc` で入力欄のフォーカスを外して normal モードに戻る。ユーザー操作起因のテキスト入力欄へのフォーカス(hint モードでの選択、マウスクリック)で自動的に insert モードへ移行する。ページ側スクリプトや `autofocus` によるフォーカスでは移行しない(自動フォーカスするサイトを開いた瞬間に insert モードへ落ちるのを防ぐ)。
- **command モード**: 画面下部のコマンドラインに `:` から始まるコマンドを入力する。`Enter` で実行、`Esc` でキャンセルし normal モードへ戻る。
  - `:open <input>`: 入力を開く。スキーム付き(`https://` 等)ならそのまま URL として開く。スキームなしでも `.` を含むホスト名形式(`example.com`)なら `https://` を、`localhost[:port]` なら `http://` を補完して URL として開く。それ以外は検索クエリとして扱い、検索エンジン(MVP では DuckDuckGo をハードコード)へ送る
  - `:quit`: 終了
- **hint モード**: `f` で発動。ページ内のクリック可能要素(リンク、ボタン、テキスト入力欄など)にキーラベルを重ねて表示し、ラベルをタイプして対象を選択(クリック相当)する。リンク選択時は遷移して normal モードへ、テキスト入力欄選択時はフォーカスして insert モードへ移行する。`Esc` でキャンセルし normal モードへ戻る。

キー配列は qutebrowser 準拠を基本方針とし、将来のタブ管理導入時に衝突するキー(`d` = タブを閉じる等)は MVP でも空けておく。キーバインドは MVP ではハードコードとし、設定ファイルによるカスタマイズは将来拡張とする。

### 3.4 MVP の完了条件

以下をもって MVP 完了と判断し、将来拡張(タブ管理)へ進む:

- 3.1〜3.3 の全項目が動作する。
- 作者が 1 週間、日常の閲覧(検索、記事閲覧、ログインが必要なサービスの利用)を owl 中心で行い、クラッシュや操作不能などの致命的な支障がない。ただしタブ不在に起因する不便は「致命的な支障」に数えない(タブは MVP 完了後の最初の拡張であるため)。

## 4. スコープ外・将来拡張

以下は MVP には含めない。優先度順に将来検討する。

| 優先度 | 機能 | 備考 |
|---|---|---|
| 高 | タブ管理 | 日常使いでは実質必須になる見込み。MVP 完了後の最初の拡張候補 |
| 高 | 設定ファイル | キーバインド・検索エンジン・起動ページのカスタマイズ |
| 中 | ブックマーク・履歴 | 履歴の永続化とコマンドラインからの補完・検索 |
| 中 | ページ内検索(`/`) | vim 風操作の自然な延長 |
| 低 | ファイルダウンロード | 保存先決定とダウンロード要求のハンドリング(MVP では非対応。3.1 参照) |
| 低 | 広告ブロック | フィルタリスト(EasyList 等)によるリクエストブロッキング |

拡張機能(WebExtensions)への対応は計画しない。

## 5. 非機能要求

- **プラットフォーム**: Linux(X11 / Wayland)。作者の環境である NixOS で動作することが必須条件。他 OS(Windows / macOS)への移植は考慮しない。
- **ビルド**: Nix flake(`nix build` / `nix develop`)で再現可能にビルドできること。WebKitGTK 等のネイティブ依存は flake で供給する。
- **起動速度**: 常用に耐える体感速度で起動すること(目安: 1 秒以内にウィンドウ表示)。
- **ミニマリズム**: UI はページ表示領域 + ステータスバー + コマンドラインのみ。ツールバーやメニューバーは持たない(エンジン既定ダイアログは例外。3.1 参照)。

## 6. 技術制約・技術選定

- **言語**: Rust(edition 2024)。
- **エンジン**: WebKitGTK。選定理由:
  - エンジンとしての組み込み実績が豊富(GNOME Web、Tauri — Rust 製アプリフレームワーク — の Linux バックエンド等)。ただし GNOME Web は C 実装、Tauri の実績は GTK3 系バインディング(`webkit2gtk` crate)のものであり、第一候補の `webkit6` crate 自体の利用実績は少ない。これが後述のフォールバックを用意する理由である。
  - Chromium 組み込み(CEF: Chromium Embedded Framework)は互換性最強だが、Rust バインディングと Nix パッケージングの難易度が高い。Servo は互換性が日常使いには不十分。実用目的とのバランスで WebKitGTK を採る。
  - JavaScript は同梱の JavaScriptCore をそのまま使い、owl 側で JS エンジンは扱わない。
- **GTK バージョン**: GTK4 + WebKitGTK 6.0 系(Rust crate: `gtk4` + `webkit6`、Nix: `webkitgtk_6_0`)を第一候補とする。バインディングの成熟度に問題が判明した場合は GTK3 + webkit2gtk-4.1 系(`webkit2gtk` crate)へフォールバックする。この決定は設計段階で確定させる。
- **GTK を直接使う理由**: wry(Tauri の WebView 抽象化レイヤ)は WebView をウィンドウに貼るだけの API であり、vim 風キーバインドに必要なキーイベントの横取り・モード管理・独自ウィジェット(ステータスバー、コマンドライン、ヒント表示)には GTK 層への直接アクセスが必要なため、wry ではなく GTK バインディングを直接使う。
- **既知のトレードオフ**: WebKitGTK は一部サイト(Chromium 前提のサービス)で互換性が劣る。日常使いで支障が大きい場合はエンジン再選定を検討するが、それはこの文書の改訂事項とする。
